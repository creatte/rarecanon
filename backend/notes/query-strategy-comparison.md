# 检索 Query 策略对比

> 待后期实验验证，当前默认保留 `rewrite_query` 节点。

## 方案 A：rewrite_query（当前）

医生输入 → LLM 改写 + 同义扩展 → retrieve

- 优势：同义扩展提高召回
- 风险：改写可能丢失语义细节，LLM 额外耗时
- 适用：知识库用词与输入有差异时

## 方案 B：直接检索

医生输入 → retrieve

- 优势：零损耗，零额外调用
- 风险：术语不一致时召回可能不足
- 适用：BGE-M3 dense 检索，输入已足够专业

## 对比维度

| | A: rewrite_query | B: 直接检索 |
|---|:---:|:---:|
| 检索耗时 | +LLM 调用 | 纯 RAG |
| 语义完整性 | 有丢失风险 | 完整保留 |
| 召回率 | 较高 | 依赖术语对齐度 |
| 噪声引入 | 可能引入无关词 | 无 |

## 结论

- 待用真实医生病例跑一批对比测试后决定
- 如果 B 召回率足够 → 去节点，省一次 LLM 调用
- 如果 B 召回有明显差距 → 保留 A，优化 prompt
