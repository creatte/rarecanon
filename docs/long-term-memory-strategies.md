# 长期记忆存储策略

三种业界主流方案，从简单到复杂。

---

## 方式一：归档时一次性总结

**当前实现用的就是这个。**

```
聊天过程                           归档时
┌─────────────┐     ┌─────────────────────────┐
│ msg1        │     │ LLM 总结最后20条消息      │
│ msg2        │     │ → 一段话摘要             │
│ ...         │     │ → 嵌入向量 → 存 PG        │
│ msg40       │     └─────────────────────────┘
└─────────────┘
```

| 优点 | 缺点 |
|------|------|
| 实现简单 | 只取最后 N 条，早期信息丢失 |
| 归档时才调一次 LLM | 40 条消息一把丢进去，超长对话 LLM 上下文不够 |

**适用场景：** 对话短且稳定的场景，早期型项目首选。

---

## 方式二：渐进式摘要

**边聊边压缩，不等到归档。**

```
聊到第10轮：摘要A = LLM(m1..m10)
聊到第20轮：摘要B = LLM(摘要A + m11..m20)
归档时：    摘要B 直接存 PG（不需要再调 LLM）
```

Redis 里同时存消息列表 + 摘要字符串 + 已摘要轮数：

```
conv:abc:messages   → [msg1, msg2, ..., msg20]
conv:abc:summary    → "患者12岁男童，DMD疑似..."
conv:abc:checkpoint → 10  ← 表示前10轮已压成摘要
```

每 10 轮触发一次：取旧摘要 + 新消息 → LLM 合并成新摘要 → 覆盖旧摘要。

```
第10轮: LLM(m1..m10)              → "12岁男童，进行性肌无力2年"
第20轮: LLM("12岁男童..." + m11..m20) → "12岁男童，DMD高度疑似，建议基因检测"
第30轮: LLM("12岁男童，DMD..." + m21..m30) → ...
```

| 优点 | 缺点 |
|------|------|
| 不丢早期信息 | 实现复杂度中等 |
| 归档瞬间完成 | LLM 调用次数增多（每10轮一次） |
| long context 对话也能搞定 | 摘要误差可能滚雪球 |

**适用场景：** 对话可能很长（20轮+）的项目。LangChain/LangGraph 内置了对这种模式的支持。

---

## 方式三：结构化提取

**不写一段话，而是填充固定字段。**

```
聊天内容
    ↓
LLM 提取结构化信息
    ↓
{
  "主诉": "进行性双下肢肌无力，2年",
  "体征": "Gower征(+)，腓肠肌假肥大未提供",
  "检查": "CK 15000 U/L",
  "疑似诊断": ["Duchenne型肌营养不良"],
  "建议检查": ["DMD基因检测", "肌电图", "肌肉活检"],
  "关键药物": [],
  "时间线": "10岁起病，12岁就诊"
}
    ↓
每个字段独立嵌入向量 → 存入 PG（或存在 metadata JSON 里）
```

检索时不是搜"一段话"，而是每个字段独立匹配：

```
用户新问题 → 向量化 → 搜"主诉"字段 → "肌无力"命中 → 召回当天那例 DMD
                     → 搜"检查"字段 → "CK"命中   → 召回另一例
```

| 优点 | 缺点 |
|------|------|
| 检索精度最高，按字段匹配 | 实现最复杂 |
| 医疗场景天然适配（病例本就有结构） | 需要精心设计 Schema |
| 不同疾病统计、对比更方便 | LLM 输出不稳定时解析失败 |

**适用场景：** 垂直领域（如医疗、法律），信息有明确结构。

### 3.1：结构化提取的时机

两种做法：

**A. 归档时提取（通用场景）**

```
会话归档 → 拿全部对话 → LLM 一次性提取 → 结构化字段入 PG
```

跟一次性总结一样，只是 LLM 输出从"一段话"变成"一段 JSON"。一次 LLM 调用的差别。

**B. 诊断节点直接产出（本项目特有优势）**

RareCanon 的 `diagnose` 节点已经输出了：

```
### 关键发现
- 症状/体征：进行性双下肢肌无力，Gower征(+)
- 检查结果：CK 15000 U/L
- 病史/病程：12岁男童，2年
```

这就是天然的结构化输出。不需要额外调 LLM，直接从 `diagnose` 结果里拆字段存：

```
diagnose 节点产出
  ↓
解析三个段落的 markdown
  ↓
{主诉: "肌无力", 体征: "Gower(+), CK 15000", 疑似: ["DMD", "BMD"]}
  ↓
存入 long_term_memories 的 metadata JSON 列
```

**零额外开销**——诊断完成顺手就存了，不增加 LLM 调用次数。

**结论：** 通用场景用 A；本项目用 B，恰好利用了 Agent 已有的结构化输出，把副产物变成了长期记忆。

---

## 选择建议

```
当前阶段        → 方案一（你已有的），先跑通
项目稳定后      → 方案二（渐进式），对话变长不丢信息
业务需要检索精度 → 方案三（结构化），医生检索历史病例
```

三者不互斥。可以先用方案一和方案二组合（渐进式摘要），后续在摘要基础上叠加方案三的字段解析。
