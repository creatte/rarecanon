# RareCanon 向量策略：为什么选择 BGE-M3

## 1. 三种检索方式对比

### 1.1 传统向量检索（Dense-only）

**做法**：用 embedding 模型把所有文档编码成高维向量，查询时计算余弦相似度。

| 优点 | 缺点 |
|------|------|
| 语义理解强，"肾上腺危象"能召回"低血容量性休克" | 医学术语精确匹配弱 |
| 不依赖分词，语言无关 | "CYP21A2 基因"可能召回"CYPA2B 基因"的相近段落 |
| 实现简单，pgvector 一条 SQL | 罕见病名、药名、剂量数字容易漏 |

**瓶颈案例**：查询 "21-羟化酶缺乏症"，dense 可能召回"17α-羟化酶缺陷症"——语义相近但完全不同的病。

### 1.2 传统 BM25（Sparse-only）

**做法**：对文档做分词，建倒排索引，查询时按词频×逆文档频率打分。

| 优点 | 缺点 |
|------|------|
| 术语精确，"21-羟化酶缺乏症"只命中包含这个词的文档 | 不会联想，"肾上腺危象"命中不了"低钠血症、高血钾" |
| 数字、药名、基因名精确匹配 | 分词质量依赖 jieba/词典，医学分词易出错 |
| 可解释，知道哪个词贡献了分数 | "21-羟化酶缺乏症"可能被切成"21"+"羟化酶"+"缺乏症" |

**瓶颈案例**：查询 "CAH 肾上腺危象处理"，BM25 只能匹配字面，无法关联到"氢化可的松替代治疗"。

### 1.3 两套系统混合（Dense + Sparse 分开部署）

**做法**：pgvector 做向量检索 + Elasticsearch/自定义倒排做 BM25，然后 RRF 融合。

| 优点 | 缺点 |
|------|------|
| 两边各取所长 | 两套系统，两份存储，两套索引 |
| 独立调优 | 分词器、索引维护、同步都是运维负担 |

## 2. BGE-M3 的解决方式

BGE-M3 一个模型同时输出两种表示：

```
输入: "21-羟化酶缺乏症怎么诊断"
              │
         BGE-M3
        ↙        ↘
   Dense          Sparse
   (1024维)      (词→权重字典)
```

### 2.1 Dense 输出（传统向量）

```
[0.023, -0.451, 0.789, ..., 0.312]    # 1024 个浮点数
```

存入 pgvector，做 ANN 检索。负责**语义泛化**——找到"意思相近"的段落。

### 2.2 Sparse 输出（替代 BM25）

```
"21-羟化酶缺乏症" → 0.85       ← 模型自动识别为核心术语，给高分
"诊断"            → 0.52
"羟化酶"          → 0.48
"缺乏"            → 0.31
"症"              → 0.05       ← 通用词自动降权
```

与传统 BM25 的关键差异：

| | 传统 BM25 | BGE-M3 Sparse |
|------|-----------|--------------|
| 分词方式 | jieba 机械切分 | 模型语义分词 |
| "21-羟化酶缺乏症" | 切为 "21" + "羟化酶" + "缺乏症" | **整体作为一个词**，权重 0.85 |
| 权重来源 | TF×IDF 统计公式 | 模型学习到的语义重要性 |
| 对数字/药名的处理 | 可能丢弃 | 自动保留并给适 

## 3. 三种方案的终态对比

| | 传统 Dense 向量 | Dense + 单独 BM25 | **BGE-M3（本项目）** |
|------|------|------|------|
| 语义理解 | ✅ | ✅ | ✅ |
| 术语精确匹配 | ❌ | ✅ | ✅ |
| 分词质量 | 不分词 | 依赖 jieba | 模型语义分词 |
| 系统数量 | 1 套 | 2 套 | **1 套** |
| 存储 | 向量 | 向量 + 倒排索引 | **仅向量** |
| 查询流程 | 查库 | 查库 + 查倒排 + RRF | 查库 → topN 上临时 sparse 重排 |
| 维护成本 | 低 | 高 | **低** |

## 4. RareCanon 的实际检索流程

```
用户查询: "21-羟化酶缺乏症怎么诊断"
              │
         BGE-M3 编码
        ↙              ↘
   dense(1024)      sparse({词:权重})
        │                  │
    pgvector            暂存内存
    余弦相似度
        │
   取 top 16
        │
        └────→ 每条候选和 sparse 词匹配 ←──┘
                     │
              加权融合: 0.7×dense + 0.3×sparse
                     │
                最终 top 8
```

## 5. 为什么 sparse 不需要存数据库

sparse 只在查询时起作用，且只对 dense 返回的 16 条候选做匹配。16 条 × 10 个词的 Python `in` 判断是毫秒级的，无需建立倒排索引。

如果每个 chunk 都存 sparse（约 1-5KB JSON），121 个病种 × 800 chunk × 3KB ≈ 额外 2.4GB 存储，收益为零。
