# 文本分块切割点选择策略：为什么从中间开始搜索？

> 本文档解释了 `_split_long_section` 中 `search_start = start + chunk_size // 2` 的设计意图，以及为什么不在整个区间 `[start, end)` 中寻找最佳断句点。

---

## 1. 背景

在处理长文本分块（chunking）时，我们需要将一篇文章按指定长度 `chunk_size` 切分成若干块，同时保证：

- 每块长度 **不超过** `chunk_size`（硬限制）
- 各块之间保留一定 `overlap`（重叠区域），防止关键信息在边界断裂
- 尽可能在 **自然语言边界**（句末、从句、空格）处切割，避免生硬截断

一个朴素的想法是：在待切割区间内 **从右向左** 搜索最高优先级的标点（如句号），用 `rfind` 找到最靠右的那个分隔符。  
这种思路看起来能最大限度地保证当前块的句子完整性。

---

## 2. 问题：完整句子 ≠ 完整语义

如果允许在整个 `[start, end)` 范围内搜索分隔符，可能出现以下情况：

**示例文本：**
```
今天天气真好。我们一起去公园吧。那里有很多花。
```
假设 `chunk_size = 14`，`start = 0`，`end = 14`。

若 `search_start = start`（即从头开始搜索），则：
- `rfind("。", 0, 14)` 会在区间 `[0, 14)` 内寻找最后一个句号。
- 索引 **6** 处有一个句号（“今天天气真好。”），这是区间内唯一的句号。
- 切割点 `cut = 6`，第一块切出：`"今天天气真好。"`（长度 7）。

表面上看，这一块是一个完整句子，语义完整。但实际上：
- 这个短句孤零零地存在，丢失了后续“去公园”和“有花”的上下文，LLM 很难仅凭这一句话理解整体语义。
- 第一块长度只有目标长度的一半，导致后续分块碎片化，上下文重叠过多，整体效率下降。
- 正因为句子太短，它反而 **破坏了语义连贯性**。

---

## 3. 方案：设置搜索下限 `search_start`

为了解决上述问题，引入一个 **切割下限**：

```python
search_start = start + chunk_size // 2
```

这意味着我们 **只在区间的后半段** 搜索切割点，前半段的所有标点都被 **故意忽略**。

### 为什么选择中间位置？

- 保证切出的块长度 **至少** 为 `chunk_size / 2`，避免产生过小的碎片块。
- 让每个块都携带 **足够的前后文**，保证语义连贯性。
- 在满足长度要求的前提下，仍然通过 `rfind` 从后往前寻找 **高优先级分隔符**，尽可能保障句子完整性。

---

## 4. 图解搜索区间

```
[ start  ...  search_start  ...  end ]
  ← 禁止搜索区 →  ← 允许搜索区（从此用 rfind）→
```

- 左半区：再完美的句号也不要，因为会导致块太小。
- 右半区：用 `rfind` 找出 **最靠右** 的合格分隔符，使得切出来的块尽量长且靠近目标尺寸。

---

## 5. 完整逻辑示例

沿用上例，但 `search_start = 0 + 14//2 = 7`：

```python
text = "今天天气真好。我们一起去公园吧。那里有很多花。"
start, end = 0, 14
search_start = 7

# 只在 [7, 14) 内搜索句号
pos = text.rfind("。", search_start, end)  # 无句号，返回 -1
# 降级寻找逗号等...
pos = text.rfind("，", search_start, end)  # 无逗号
pos = text.rfind(" ", search_start, end)   # 索引 9 有空格 (吧。那里)
# 最终可能退化为硬切，或者找到较低级别分隔符
```

虽然这个例子在右半区没有句号，但算法会逐级降级，**最终在保证块长足够的前提下，找到当前能用的最好分隔符**。  
即使被迫在空格处硬切，每个块依然保留了约一半的目标长度，上下文信息远优于一个孤立的短句。

---

## 6. 降级顺序说明

代码中的分隔符优先级：

1. `。！？\n` —— **句子结尾**（最理想）
2. `，,；;：:` —— **从句、并列边界**（次理想）
3. ` 　` —— **空格**（最后的语义边界）
4. 如果全部找不到 → **硬切**（直接取 `end` 位置）

搜索只在右半区进行，每级遍历所有对应标点，取 **最靠右** 的一个。  
找到后立即结束降级，避免使用更低级的分隔符。

---

## 7. 总结

| 搜索策略 | 优点 | 缺点 |
|----------|------|------|
| 从 `start` 开始搜索 | 容易找到完整句子 | 易产生过小块，破坏整体上下文，碎片化 |
| 从 `search_start` 开始搜索 | 块大小均匀，每块都有足够上下文，语义连贯性更好 | 偶尔会在从句或空格处切断句子（但重叠可补救） |

**设计核心**：  
不是追求每一个块的“局部句子完整”，而是 **在全局长度约束下，让每一个块都成为一个信息充分、上下文连贯的语义单元**。  
`search_start` 正是为了阻止算法被“局部完美句号”诱惑，而放弃整体语义质量。

---

> 此策略广泛应用于 RAG、长文本 LLM 推理等场景的分块预处理中。
